# alpha191-quant Agent Guide

本文件记录本项目长期有效的协作、文档、数据和实施规则。阶段性状态、临时调研日志、一次性待办不应写入本文件。

## 语言与沟通

- 面向用户的回复、待办、文档和状态更新默认使用中文。
- 技术术语可保留英文原词，但需要保证上下文清晰。
- 涉及交易、收益、风险时必须表达为工程与研究结论，不做投资承诺。

## 主规格入口

- 当前项目主规格是 `docs/alpha191-quant-system-spec.html`。
- 后续 `/goal`、实现、验收、复盘都应先读取本文件，再读取相关源码和数据源说明。
- 当需求、设计、里程碑、验收标准发生实质变化时，优先更新这个 HTML artifact，而不是新建零散 Markdown 文档。
- Markdown 只用于项目长期规则、极短索引或用户明确要求的文档；不要把阶段性说明散落成多个 Markdown 文件。

## HTML Artifact 文档约定

落实 Thariq 在 Claude Code HTML artifact 文章中的文档观点：复杂需求不只写线性文字，应把需求、设计、交互、实施路径和验收标准做成可视化、可点击、可对照的 HTML artifact。

本项目的 human-facing 文档应满足：

- 自包含：HTML/CSS/JS 尽量写在单个文件内，避免依赖外部服务。
- 可对照：需求、数据流、UI 原型、milestones、验收命令要能在同一文档里快速跳转。
- 可交接：包含明确的 `/goal` 合约，让 Codex 能按文档连续实施。
- 可验收：每个 milestone 必须有输入、产物、验证方式和停止条件。
- 可反馈：UI 原型要足够具体，便于用户直接基于界面讨论需求，而不是只读长文档。
- UI 原型必须是独立完整页面，不能退回文档内局部 mock；当前工作台原型入口是 `docs/alpha191-workbench-prototype.html`。

## 项目边界

- 本项目目标是 A 股量化研究与手动交易信号系统：每天给出候选标的、方向、建议成交价格区间、风控说明和数据依据。
- 当前不实现程序化下单，不接券商交易接口，不自动发送真实交易指令。
- 先建设数据底座、首批 Alpha191 简单因子、因子评估、轻量回测和可视化 UI，再扩展更复杂策略。
- 不要为了未来假设提前搭过度抽象；优先保证数据可得、口径可查、结果可复现。

## 数据源规则

- Tushare 是日频行情、复权、基础信息、基本面和部分分钟数据的重要来源，token 从 `.env` 读取，严禁打印、提交或写入文档。
- 通达信数据源是日内行情的重要来源，优先研究并支持 `pytdx`、`mootdx`；特别关注 `1m`、`5m`、`15m`、`30m`、`60m` 分钟 K 线。
- 东方财富、腾讯财经、巨潮资讯、同花顺等公开来源可作为行情、公告、新闻、资金流、题材热度和 fallback 数据源。
- 任一数据字段进入核心表或信号输出前，必须记录来源、刷新时间、字段单位、复权状态和失败降级路径。
- 日内行情默认不复权，只用于盘中节奏、价格区间和执行参考；因子研究与回测需明确是否使用复权价格。

## 因子与研究规则

- 因子应按 `量价因子`、`基本面因子`、`另类因子` 分类组织。
- Alpha191 公式必须保留公式来源、依赖字段、计算窗口、复权要求和方向判定方式。
- 因子好坏不能只看单日表现，至少需要覆盖率、Rank IC、ICIR、分组收益、换手率、缺失率和样本外表现。
- 首批实现优先选择字段简单、窗口短、依赖清晰的 Alpha191 因子，并把评估流程跑通。
- 回测先做研究级验证，不把回测收益当作实盘承诺。

## 实施与验收

- 实施顺序遵循 HTML 主规格中的 M0-M6 milestones。
- 每个 milestone 完成后必须运行对应验证命令，并把失败原因收敛到可修复项。
- UI 必须能展示行情、数据健康、因子评估、回测结果和每日信号；不要只做静态说明页。
- 任何生成的交易信号都必须附带数据来源、生成时间、价格区间、失效条件和风险提示。

## 安全

- 不读取或输出 `.env` 中的 secret 值；只允许验证变量是否存在。
- 不在仓库内保存 token、cookie、账号、交易密码或券商凭据。
- 不执行递归删除、跨项目写入或系统目录操作，除非用户明确确认。
